# 프로젝트 개요 및 담당 파트

## 1) 프로젝트 개요

### 한 줄 소개

**알맹이(Almaengi)**는 소상공인/자영업자의 인사·노무 운영(근로계약, 출결, 급여 정산, 사내 커뮤니케이션, 노동법 질의)을 하나의 서비스로 통합한 올인원 플랫폼입니다.

### 개발 배경

- 소규모 사업장은 근로계약 관리, 출결 확인, 급여 계산, 공지/소통, 법률 질의가 각각 분리되어 운영 비용이 큼
- 근로기준법 반영 급여 계산과 문서 발급, 실시간 커뮤니케이션, 노무 상담까지 한 흐름에서 처리할 필요가 있었음

### 핵심 기능

- 전자 근로계약 작성/서명 및 PDF 발급
- QR/GPS 기반 출퇴근 인증 및 스케줄 관리
- 주휴/연장/야간/공제 반영 급여 자동 정산
- 급여명세서 자동 생성 및 조회
- 매장 내 실시간 채팅 및 푸시 알림
- 노동법 특화 AI 챗봇(RAG 기반)

### 시스템 구성

- **Frontend**: React + TypeScript + Vite (PWA/FCM)
- **Backend**: Spring Boot + JPA + Redis + JWT + WebSocket(STOMP)
- **AI**: FastAPI + LangChain + Qdrant (RAG)
- **Infra**: Docker Compose + Nginx + Jenkins
- **DB**: PostgreSQL

### 아키텍처 특징

- 도메인 단위 모듈 분리(`attendance`, `auction`, `chat`, `notification`, `payroll`, `store` 등)
- JWT 기반 인증/인가 + Redis 활용(토큰/실시간 상태/메시지 fan-out)
- STOMP WebSocket + Redis Pub/Sub 기반 실시간 확장 구조
- AI 질의 라우팅(일반 응답/RAG/Agent)으로 정확도와 활용성 확보

### 기술적 강점

- 출결-급여-문서-알림-AI 상담까지 실제 업무 흐름을 끊김 없이 연결
- 실시간 기능(채팅/경매 이벤트)과 운영 안정성(캐시/예외/권한)이 함께 설계됨
- 로컬~운영 환경(컨테이너/프록시/CI)이 일관된 구조로 준비됨

## 2) 프로젝트에서 담당한 파트

### 역할 요약

프로젝트에서 저는 **백엔드 중심의 핵심 도메인 설계 및 구현**을 담당했고, 실시간/알림 기능은 프론트엔드까지 연결해 **서비스 동작의 완결성**을 높였습니다.

  - 총 54개(비-머지 30개)
  - BE 중심 + BE/FE 크로스 레이어 작업 다수

### 담당 영역

#### A. 백엔드 기반 아키텍처 구축

- 공통 응답 포맷, 전역 예외 처리, 에러코드 체계 정립
- JPA Auditing, Security/Swagger 초기 설정
- `@AuthUser` 기반 인증 사용자 주입 패턴 구축(하드코딩 userId 제거)

**성과**
- 이후 도메인 개발 시 일관된 API/예외/인증 패턴을 재사용할 수 있는 기반 확보

#### B. 매장/직원/스케줄 도메인

- 매장 CRUD, 직원 초대 코드 발급/합류/승인/퇴사 흐름 구현
- `WAITING -> WORKING` 상태 전이 중심으로 권한 정책 반영
- 직원 본인 관점 API(소속 매장 조회, 본인 출결 조회) 추가

**성과**
- 단순 CRUD가 아닌 운영 상태 기반 도메인 정책을 구현해 실제 업무 시나리오 대응력 강화

#### C. 경매(대타) 도메인

- 경매 등록/입찰/마감/수정/취소/상세/목록 구현
- 낙찰 시 출결(Attendance) 생성 및 예정 근무시간 반영
- 경매 인사이트 리포트(성공률/평균 낙찰가/타임라인) 구현
- 경매 이벤트 실시간 반영(WebSocket 구독 채널 연동)

**성과**
- 경매 도메인을 단일 기능이 아닌 "비즈니스 규칙 + 통계 + 실시간 동기화"까지 확장

#### D. 채팅 및 AI 챗봇 연계

- DM/GROUP/BOT 채팅 구조, 메시지/읽음 처리 로직 구현
- STOMP + Redis Pub/Sub 기반 채팅 fan-out 구조 구현
- 챗봇 연동(`/ask`) 및 비동기 응답 저장, fallback 처리 구현
- 매장 생성/직원 승인 시 BOT 방 자동 생성 및 웰컴 메시지 처리

**성과**
- 멀티 인스턴스에서도 확장 가능한 실시간 채팅 구조를 확보하고, 챗봇 경험을 서비스 플로우에 자연스럽게 통합

#### E. FCM 알림 체계 (BE + FE)

- BE: 토큰 등록, 알림 저장, 읽음 처리, push-only 전송 분리 설계
- FE: 권한 요청 -> SW 등록 -> 토큰 동기화 -> foreground 리스너 흐름 구현
- 멀티탭/중복 실행/AbortError 상황에 대한 방어 로직 구현
- 중복 알림 이슈 수정(data-only 기반 + 단일 표시 전략)

**성과**
- 알림 기능을 "보내는 기능" 수준이 아니라 운영 환경의 실패/중복까지 고려한 안정적 구조로 고도화

#### F. 외부 연동 및 운영 보강

- 카카오 지오코딩 연동(주소 -> 위/경도 변환) 및 예외 매핑
- PWA 설정(vite-plugin-pwa)과 nginx 캐시 정책 연동

**성과**
- 사용자 편의 기능(위치 기반)과 운영 반영성(PWA/SW 배포 안정화) 동시 확보

## 3) 기술적 특장점 (내 기여 관점)

- **정합성 우선 실시간 설계**: DB 커밋 이후 이벤트 발행(`afterCommit`)으로 롤백 데이터의 선노출 방지
- **운영 리스크 대응형 구현**: 푸시 실패, 외부 API 장애, 토큰 race condition을 기능 차단 없이 처리
- **상태 기반 권한 제어**: 직원 상태(`WAITING/INVITED/WORKING/RESIGNED`)를 도메인 전반 정책에 반영
- **도메인 확장성 확보**: 공통 응답/예외/인증 레일 위에 기능을 쌓아 빠른 확장 가능

## 4) 리스크 관리 사례

- **실시간 이벤트 정합성 리스크**
  - 위험: 트랜잭션 롤백 시 잘못된 이벤트가 먼저 전파될 수 있음
  - 대응: 경매/채팅 이벤트를 `afterCommit`에서만 발행

- **알림 중복/실패 리스크**
  - 위험: 백그라운드+포그라운드 중복 노출, 토큰 중복 발급, 푸시 실패
  - 대응: data-only 메시지, 리스너 중복 등록 방지, cross-tab lock/cooldown, 실패 시 best-effort

- **권한 오남용 리스크**
  - 위험: 승인 전 직원의 도메인 기능 접근
  - 대응: 상태 기반 접근 제한과 컨트롤러/서비스 계층 권한 검증 강화

- **외부 연동 불안정 리스크**
  - 위험: 지오코딩/챗봇 외부 API 오류로 핵심 흐름 중단
  - 대응: 도메인 에러코드 매핑 + fallback 처리로 서비스 연속성 유지
