# 프로젝트 경험 (이력서형)

## 1. 프로젝트 개요

- **프로젝트명**: 알맹이(Almaengi)
- **유형**: 팀 프로젝트 (소상공인 인사/노무 운영 통합 서비스)
- **기간**: 2026.02 ~ 2026.04 (커밋 기준)
- **목표**: 근로계약, 출결, 급여, 사내 소통, 노동법 질의응답을 하나의 서비스로 통합
- **담당 범위**: 백엔드 핵심 도메인 설계/구현 중심 + 실시간/알림 기능의 프론트 연동

## 2. 기술 환경

- **Frontend**: React, TypeScript, Vite, React Query, Zustand, Tailwind CSS, PWA, Firebase Cloud Messaging
- **Backend**: Java 17, Spring Boot, Spring Security, JPA(Hibernate), JWT, Redis, STOMP(WebSocket)
- **AI**: FastAPI, LangChain, Qdrant (RAG 기반 질의응답)
- **Database**: PostgreSQL
- **Infra/DevOps**: Docker Compose, Nginx, Jenkins
- **협업/버전관리**: Git, GitHub (커밋 기반 기능 단위 개발)

## 3. 담당 역할

- 공통 API 응답/예외/인증 주입(`@AuthUser`) 등 **백엔드 표준 레일** 구축
- **매장/직원/스케줄** 도메인 구현 (초대코드, 합류/승인 상태 전이, 권한 정책)
- **경매(대타)** 도메인 구현 (등록/입찰/낙찰/마감/리포트, 실시간 반영)
- **채팅(DM/GROUP/BOT)** 도메인 및 WebSocket + Redis Pub/Sub 구조 구현
- BE-AL 서비스 연동(챗봇 `/ask`) 및 비동기 응답 저장/fallback 처리
- **FCM 알림 E2E** 구현 및 중복/경쟁 상태 안정화(토큰 동기화, SW 라우팅)
- 카카오 지오코딩, PWA, nginx 캐시 정책 등 운영 보강

## 4. 주요 기능 및 구현 과정

### 4-1. 실시간 경매 동기화

- **문제**: 경매 상태 변경(등록/수정/마감)이 사용자 화면에 즉시 반영되지 않음
- **구현**:
  - 백엔드에서 경매 이벤트 발행 채널 구성
  - 트랜잭션 커밋 이후(`afterCommit`) 이벤트 발행으로 정합성 보장
  - 프론트에서 구독 후 React Query 캐시 무효화로 화면 즉시 갱신
- **결과**: 새로고침 없이 실시간 동기화되는 사용자 경험 제공

### 4-2. 채팅 + 챗봇 통합

- **문제**: 실시간 채팅과 챗봇 응답을 같은 흐름에서 안정적으로 처리해야 함
- **구현**:
  - DM/GROUP/BOT 채팅방 구조 및 읽음 포인터 로직 구현
  - STOMP + Redis Pub/Sub로 멀티 인스턴스 fan-out 지원
  - 챗봇 호출을 비동기로 처리하고 실패 시 fallback 메시지 제공
- **결과**: 채팅 지연 체감 감소 및 챗봇 장애 시에도 대화 흐름 유지

### 4-3. FCM 알림 안정화

- **문제**: 멀티탭/브라우저 환경에서 토큰 중복 등록 및 중복 알림 발생
- **구현**:
  - 토큰 동기화에 in-flight lock, cross-tab lock, cooldown 적용
  - SW/foreground 처리 경계를 명확히 분리하여 중복 표시 방지
  - push 실패는 best-effort로 처리하여 핵심 비즈니스 흐름 보호
- **결과**: 알림 신뢰도 향상 및 운영 중 장애 전파 최소화

### 4-4. 상태 기반 권한 정책

- **문제**: 승인 전 직원(WAITING/INVITED)의 기능 오남용 가능성
- **구현**:
  - 직원 상태 전이 모델링(WAITING -> WORKING)
  - 출결/채팅/경매/계약 등 도메인에서 상태 조건 일관 검증
- **결과**: 도메인별 권한 분기 일관성 확보 및 정책 누락 리스크 감소

## 5. 성과 및 배운 점

### 성과

- 백엔드 중심으로 핵심 도메인을 설계/구현하고 프론트 연동까지 완료하여 기능 완성도 향상
- 실시간/알림 영역에서 정합성과 안정성을 함께 고려한 구조를 도입
- 개인 기여 커밋: **54개(비-머지 30개)**

### 배운 점

- 실시간 시스템에서는 "빠른 전파"보다 **트랜잭션 경계와 정합성 보장**이 우선임을 체득
- 장애가 발생하더라도 서비스 핵심 흐름을 유지하는 **best-effort 설계**의 중요성 학습
- 권한 정책은 기능별 조건문이 아니라 **상태 모델 중심**으로 설계해야 유지보수가 쉬움
- 테스트/시나리오 기반 검증이 리팩토링과 운영 안정성의 핵심임을 확인
