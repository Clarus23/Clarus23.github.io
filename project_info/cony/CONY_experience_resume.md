# 프로젝트 경험 (이력서형) - CONY

## 프로젝트 개요
- **프로젝트명**: CONY (기프티콘 통합 관리 서비스)
- **진행 시기**: 2026.01 ~ 2026.02 (집중 개발 기간 기준)
- **프로젝트 소개**: OCR 기반 기프티콘 등록, 위치 기반 알림(Geofence), 거래/결제, 추천 기능을 제공하는 모바일 중심 서비스
- **핵심 목표**: 기프티콘 만료/미사용 문제를 줄이고, 실사용 전환율을 높이는 사용자 경험 제공

## 기술 환경
- **Client**: React Native(TypeScript), Next.js
- **Backend**: Java 17, Spring Boot, Spring Security, Spring Data JPA, Spring Batch
- **AI**: Python, FastAPI (OCR/추천)
- **Data/Infra**: MySQL, Redis(GEO), AWS S3, Docker Compose, Jenkins, Prometheus, Grafana
- **External**: Google/Kakao OAuth, KakaoPay, Firebase Cloud Messaging(FCM)

## 담당 역할
- Manage/Payment 중심의 **백엔드 핵심 기능 설계 및 구현**
- 기프티콘 도메인, Geofence, 알림, 추천, 결제 연동 정합성, 내부 API 보안 담당
- 커밋 기준 기여: `정혜원`, `Hyewon_Jung` 합산 **244건** (전체 918건 중 약 26.6%)

## 주요 기능 및 구현 과정

### 1) 기프티콘 관리 도메인 고도화
- 문제: 조회 조건 확대와 상태 변화(사용/취소/삭제)가 누적되며 로직 복잡도 증가
- 구현:
  - JPA Specification 기반 동적 검색(브랜드/상품명/카테고리/만료임박/위치기반)
  - soft delete 및 권한/예외 처리 강화
  - 사용/사용취소/이력 관리 로직 안정화
- 결과: 기능 확장 시 조건 추가가 용이한 구조로 전환

### 2) Geofence 성능 최적화 및 위치 알림 구현
- 문제: 위치 기반 조회를 DB 중심으로 처리 시 응답 지연과 부하 위험
- 구현:
  - Redis GEO + Hash + Pipeline으로 주변 매장 조회 구조 설계
  - Spring Batch로 매장 캐시 워밍업 자동화
  - Android 지오펜스 제한 대응 클러스터링 로직(근접 매장 묶음 처리) 적용
  - 브랜드 단위 쿨다운으로 중복 알림 완화
- 결과: 위치 기능의 응답성과 운영 안정성 개선, 알림 피로도 감소

### 3) 알림 시스템(FCM + 스케줄러) 운영 안정화
- 문제: 푸시 전송 실패 시 사용자/운영 측 추적이 어려움
- 구현:
  - Geofence/만료/자동판매 알림 공통 발송 구조 구성
  - FCM 전송 실패 시에도 알림 히스토리 저장
  - 만료 임박 알림 스케줄러(매일 오전 10시) 구현
- 결과: 장애 상황에서도 이벤트 이력 추적 가능, 운영 대응성 향상

### 4) MSA 간 정합성 및 내부 API 보안 강화
- 문제: 판매/구매 이벤트 시 payment-manage 간 상태 불일치 위험
- 구현:
  - 판매 시작/취소/구매 완료 시 상태/소유권 동기화 API 구현
  - `X-Internal-Key` 기반 내부 API 인터셉터 인증 경계 추가
  - 내부 통신 테스트 코드 작성으로 회귀 리스크 관리
- 결과: 도메인 정합성과 서비스 간 호출 신뢰성 강화

### 5) OCR/추천 연동 고도화
- 문제: 외부 연동 시 DTO 포맷 불일치 및 추천 실패 대응 필요
- 구현:
  - OCR 요청 snake_case 매핑 이슈 해결
  - 일반 추천(거리/유효기간) + AI 개인화 추천 fallback 전략 구현
- 결과: 연동 안정성과 사용자 추천 지속성 확보

## 성과 및 배운점

### 성과
- 백엔드 핵심 기능을 end-to-end로 구현하며 서비스 핵심 가치(등록-알림-거래-추천)를 연결
- Redis 기반 위치 기능 최적화와 내부 API 동기화/보안 설계로 운영 안정성 개선
- 이슈 대응 중심의 다수 `fix`/`test` 커밋을 통해 품질 안정화 주도

### 배운점
- **도메인 정합성**은 기능 구현보다 우선 설계해야 운영 장애를 줄일 수 있음을 체감
- 외부 연동은 성공 시나리오보다 **실패 시나리오(fallback/히스토리 보존)** 설계가 핵심임을 학습
- 성능 최적화는 인프라 확장보다 먼저 **조회 경로 재설계(캐시/파이프라인/배치)**로 해결할 수 있음을 경험
